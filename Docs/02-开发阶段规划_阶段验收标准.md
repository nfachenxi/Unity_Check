# 开发阶段规划 + 阶段验收标准

## 第一阶段：项目启动与基础链路打通
### 要实现的功能
- 搭建基础工程与运行环境（后端、任务队列、数据库）。
- 接收 Git Webhook 事件并入库。
- 建立基础任务流转（接收事件后可触发评估任务）。
- 支持 `push` 与 `pull_request` 两类事件。
- 通过 `.env` 统一管理数据库、缓存与模型连接参数。

### 预期效果
- 代码仓库发生提交或 PR 事件后，系统可以自动感知并记录。
- 后端能够稳定触发并执行最小任务流程。

### 验收标准
- 至少 1 类 Git 事件可稳定接入并落库。
- 任务可被成功消费并返回处理状态。
- 基础服务可一键启动并正常联通。
- `push` 与 `pull_request` 均可触发入库和任务消费。
- 任务处理结果可在数据库中追踪（状态、风险等级、错误信息）。

### 第一阶段实施细则（本轮确定）
- 事件入口：`POST /webhook/github`
- 健康检查：`GET /health`
- 事件查询：`GET /events/latest`
- 入库表：`github_events`
- 异步执行：Celery worker 消费 `unity_check.process_github_event`
- 模型调用：优先使用 DeepSeek `deepseek-chat`，失败时回退为可追踪错误信息

## 第二阶段：规则检测与结果结构化
### 要实现的功能
- 集成 Unity/C# 静态规则检测能力。
- 输出统一格式的检测结果（问题类型、位置、严重级别）。
- 建立基础评分规则（如严重级别加权）。

### 预期效果
- 系统可对代码变更给出可读、可定位的问题清单。
- 结果可用于后续大模型评估与通知判断。

### 验收标准
- 能对示例提交稳定输出规则检测结果。
- 输出格式统一，且可被后续模块直接消费。
- 至少具备基础分级能力（高/中/低风险）。

## 第三阶段：大模型多轮评估
### 要实现的功能
- 接入大模型接口，基于变更内容进行语义评估。
- 实现多轮评估流程（规则结果 + 语义评估 + 复核总结）。
- 生成可执行建议（风险说明、修改建议、优先级）。

### 预期效果
- 除规则命中外，系统还能识别潜在架构风险与维护风险。
- 输出结果更贴近代码审查场景。

### 验收标准
- 对同一提交可完成完整多轮评估并产出总结结果。
- 评估结果包含风险等级与建议动作。
- 处理流程可复现，失败任务可追踪。

## 第四阶段：通知联动与可视化展示
### 要实现的功能
- 按阈值触发通知（企业微信或飞书）。
- 提供基础看板（提交记录、评分趋势、问题分布）。
- 支持查看单次评估详情。

### 预期效果
- 高风险提交可被及时提醒，形成协作闭环。
- 指导团队观察质量趋势与高频问题类型。

### 验收标准
- 通知可按规则稳定触发且内容完整。
- 看板可查询历史记录并展示核心指标。
- 单次评估详情可追溯到具体提交。

## 第五阶段：联调优化与答辩准备
### 要实现的功能
- 完成全链路联调与稳定性修正。
- 补齐演示数据与对比结果（前后效果）。
- 输出答辩演示流程与关键指标说明。

### 预期效果
- 系统可进行连续演示，不出现关键链路中断。
- 能清晰说明系统价值、效果和边界。

### 验收标准
- 端到端流程可稳定跑通多次。
- 具备可展示的质量改进数据或案例。
- 答辩演示流程完整，时间可控。
