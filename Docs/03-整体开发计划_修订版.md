      ▼
[编排器任务] ──┬──► [Git Service: clone/fetch/diff]
               │        │
               │        ▼

               │        │
               │        ▼

## Phase 1b：Git 操作 + Diff 接入 + Docker（预估 3-4 天）

**依赖：** Phase 1a
**核心目标：** 补齐 Git clone/fetch/diff 能力，LLM 评估接入实际代码，Docker Compose 一键启动

### 1b.1 数据模型变更
`github_events` 表新增列（扩展，不改现有列）：
```
after_sha    VARCHAR(40)   -- push/PR 后的 commit SHA
before_sha   VARCHAR(40)   -- push/PR 前的 commit SHA
clone_path   VARCHAR(512)  -- 本地 clone 路径
diff_content TEXT          -- 提取的 diff 内容
diff_size    INTEGER       -- diff 字节数，用于 LLM 上下文窗口守卫
```
现有列保持不变。

### 1b.2 核心功能
1. **Git Service**（`src/unity_check/git_service.py`，GitPython 库）— clone/fetch/diff，使用 `GIT_SSH_KEY_PATH` 配置 SSH 认证
2. **LLM 评估接入实际 diff** — 修改 `llm.py` prompt，加入 diff 内容（含截断守卫，如 8000 字符）
3. **Docker Compose** — api / worker / redis / db 四个服务 + `Dockerfile`
4. **Webhook 解析增强** — 从 payload 中提取 before/after SHA

### 1b.3 API/端点
- `GET /health`（已有）
- `POST /webhook/github`（已有）
- `GET /events/latest`（已有）
- `GET /events/{event_id}` — 新增：单个事件详情含 diff

### 1b.4 验收标准
1. 模拟 push webhook → clone/fetch → diff → 入库 → LLM 评估含实际代码
2. `docker compose up` 启动全部 4 个服务，`/health` 正常
3. `GET /events/{id}` 返回完整事件含 diff_content
4. Git Service 模块测试覆盖率 >= 70%

## Phase 2：静态规则检测 — Roslyn 容器化（预估 6-8 天）

**依赖：** Phase 1b（Git diff 可正常工作）
**核心目标：** Roslyn 作为 Docker 化 API 被 Python 后端调用，输出结构化规则结果

### 2.1 架构：Roslyn 作为 Docker Sidecar

```
[Python Celery Worker]
      │ HTTP POST (JSON)
      ▼
[Roslyn Analyzer API 容器]
   - ASP.NET Core 8 Minimal API
   - Microsoft.CodeAnalysis + StyleCop.Analyzers
   - 接口：POST /analyze  { files: [{path, content}] }
   - 返回：[{ file, line, severity, rule_id, message }]
```

### 2.2 新增表
```sql
rule_results (
    id, event_id FK, round INTEGER DEFAULT 1,
    file_path, line_number, column_number,
    rule_id, rule_name, severity, category, message, suggestion,
    raw_output JSONB, created_at
)
```

### 2.3 核心功能
- `roslyn-analyzer/` 目录存放 .NET 8 项目 + Dockerfile
- `rule_service.py` — 调用 Roslyn API + 解析结果 + 评分
- 编排任务中新增第 1 轮（规则检测）

### 2.4 新增端点
- `GET /events/{id}/rules`
- `GET /rules/stats`

### 2.5 验收标准
1. `roslyn` 服务可被 `POST /analyze` 调用
2. push webhook → 规则分析 → `rule_results` 表中 >=1 条记录
3. 至少 10 条规则活跃产生结果

## Phase 3：多轮 LLM 评估（预估 8-10 天）

**依赖：** Phase 2（规则结果数据必须存在）
**核心目标：** 三轮评估流程完整跑通，每轮结果独立持久化

### 3.1 三轮评估定义

```
Round 1：规则检测（自动化，Phase 2）
  输入：code diff
  输出：rule_results（结构化违规）
  耗时：<10s，确定性

Round 2：LLM 语义评估
  输入：code diff + Round 1 规则结果
  输出：语义发现（架构风险、设计模式误用、Unity 反模式、性能陷阱等）
  耗时：10-30s

Round 3：LLM 复核总结与评分
  输入：code diff + Round 1 结果 + Round 2 发现
  输出：overall_score、risk_level、executive_summary、top_issues、recommendation、action_items
  耗时：10-20s
```

### 3.2 新增表
```sql
evaluation_rounds (
    id, event_id FK, round_number, round_type, status,
    input_summary JSONB, output_data JSONB, score FLOAT,
    model_name, tokens_used, duration_ms,
    error_message, started_at, completed_at, created_at
)
```

`github_events` 新增（Round 3 填充）：
```
overall_score FLOAT
final_risk_level VARCHAR(16)
recommendation VARCHAR(32)    -- merge_ready / needs_review / blocked
executive_summary TEXT
```

### 3.3 核心功能
- `orchestrator.py` — 三轮编排引擎
- `llm.py` 重构 — 拆分为 `semantic_review()` 和 `synthesis_summary()`
- Token 计数 + 重试机制（3 次指数退避）
- Prompt 模板化（文件或常量管理）

### 3.4 新增端点
- `GET /events/{id}/evaluations`
- `GET /events/{id}/assessment`
- `POST /events/{id}/re-evaluate`

### 3.5 验收标准
1. 单次 push → 3 轮顺序执行 → `evaluation_rounds` 表 3 行
2. Round 2 输出中至少包含 1 条不在 Round 1 规则结果中的发现
3. Round 3 输出包含完整字段：score、risk_level、summary、top_issues、recommendation、action_items
4. Round 2 失败时 Round 3 仍执行并标注缺失数据
5. Token 用量记录在案
6. 典型 diff（<500 行）60 秒内完成全流程

## Phase 4：通知 + 可视化看板（预估 10-12 天）

**依赖：** Phase 3（最终评估数据必须存在）
**核心目标：** 通知消息构建与入库（实际发送由独立工具平台实现）+ Vue3 看板 4 页

### 4.1 新增表
```sql
notifications (
    id, event_id FK, channel, trigger_reason,
    risk_level, message_content, webhook_url,
    status, sent_at, error_message, created_at
)
```

### 4.2 通知服务（简化版）
- **我们只负责**：阈值判断 + 消息内容构建（Markdown/JSON）+ 通知记录入库
- **独立工具平台负责**：实际发送企业微信/飞书消息、发送状态回调
- 阈值规则：critical/high 一定生成通知、medium + score<70 生成通知、low 不生成
- 消息模板：企业微信 markdown 格式、飞书交互式卡片格式
- 配置：`NOTIFY_RISK_THRESHOLD`、`NOTIFY_SCORE_THRESHOLD`
- 预留 API：`POST /api/notifications/{id}/send-status` 供工具平台回调更新发送状态

### 4.3 前端（`frontend/` 目录，Vue3 + Element Plus + Vite）

| 页面 | 路由 | 内容 |
|------|------|------|
| 概览看板 | `/` | 提交趋势折线图、风险分布饼图、问题类型柱状图、状态汇总、最近事件表 |
| 事件列表 | `/events` | 分页表格 + 筛选（日期、类型、风险、状态） |
| 事件详情 | `/events/:id` | 元信息 + Round 1/2/3 分页签 + Diff 视图 |
| 统计中心 | `/stats` | 日期范围选择 + 评分趋势 + 高频规则 + 文件热点 |

### 4.4 新增 API
- `GET /api/dashboard/summary`
- `GET /api/dashboard/trends?days=30`
- `GET /api/dashboard/issue-distribution`
- `GET /api/events?page=&filters=...`
- `GET /api/events/{id}`
- `GET /api/stats/scores?from=&to=`
- `GET /api/stats/hotspots?limit=10`
- `GET /api/notifications?event_id=`

### 4.5 验收标准
1. high/critical 事件触发通知并入库
2. low 风险不触发通知
3. 看板概览页 2 秒内加载并显示 4 个图表
4. 事件列表支持筛选
5. 事件详情 4 个分页签数据完整
6. 统计页日期选择可用
